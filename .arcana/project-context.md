## Testing
- Test runner: Vitest 4.1.9 (jsdom env, `@testing-library/react` + `jest-dom`)
- Mutation testing: Stryker 9.6.1 (`@stryker-mutator/vitest-runner`) ‚Äî `npm run mutate` (one root config, `stryker.config.json`)
- E2E runner: CodeceptJS 4.0.8 + Playwright 1.61 ‚Äî `npm run test:e2e` (via `start-server-and-test`)
- Network mocking: MSW 2.14.6
- Coverage: `@vitest/coverage-v8`
- Test directory: co-located `**/*.test.ts(x)`; e2e specs in `apps/web/src/test/**/*.e2e.ts`
- Root commands: `npm test` (vitest run), `npm run test:watch`, `npm run test:e2e`, `npm run mutate`
- Mutation testing runs from **one root config** (`stryker.config.json`) over the whole vitest workspace. Scope: `apps/web/src/**/model|lib/**` + `apps/api/src/**` + `packages/shared/src/**`. `packages/ui` and web `ui/` components are deliberately excluded (they are covered behaviorally, per the testing strategy).
- **Always scope a run to the ticket's 2‚Äì3 files** (`npx stryker run --mutate '<path>'`), never bare `npm run mutate`. A full sweep is ~580 mutants and will not finish in a useful time; a scoped run on the file that holds the ticket's rules is ~2‚Äì3 minutes. This is the single most-violated rule here ‚Äî an unscoped sweep is what makes the phase look worthless.
- **Derive the scope from git instead of hand-listing files.** Compute the branch's changed files and feed them to `--mutate` (bash):
  ```bash
  FILES=$(git diff --name-only --diff-filter=d main...HEAD \
    | grep -E '^(packages/shared/src|apps/api/src|apps/web/src/.*(model|lib))/.*\.ts$' \
    | grep -vE '\.(test|spec|gen|d)\.ts$' | paste -sd,)
  npx stryker run --mutate "$FILES"
  ```
  `main...HEAD` (three dots) diffs from the branch point, so the run covers exactly what the branch added. This is StrykerJS's equivalent of .NET's `--diff` ‚Äî **StrykerJS has no `--since` flag**. `incremental: true` (already set) is the complementary auto-diff: it reuses the prior report so repeat runs re-test only changed code/tests. Use the git recipe for a deterministic branch-scoped run; incremental for cheap local re-runs.
- **`vitest.related: true` is load-bearing, not an optimization.** Vitest walks the import graph and runs only the tests that actually reach the mutated file ‚Äî *including tests in other workspaces*. Measured on `packages/shared/src/validation-schemas/resident.ts` (148 mutants): shared's own suite alone ‚Üí 71.62%, 38 survivors, 29s; whole monorepo suite (`related: false`) ‚Üí 87.16%, 15 survivors, **23m33s**; `related: true` ‚Üí 88.51%, 13 survivors, **2m31s**. Same signal as the exhaustive run, ~9√ó faster.
- **Business rules live in `packages/shared`, not in `apps/web`.** `apps/web/src/**/lib/*` holds thin re-exports (`carrier-warning.ts`, `phone.ts`); every AC-critical rule ‚Äî apartment ranges, name bounds, phone validity, E.164 normalization, carrier prefixes ‚Äî is in `packages/shared/src/validation-schemas/resident.ts`. Point a mutation run at the rule, not at its adapter.
- **In a monorepo a survivor is a lead, not a verdict** ‚Äî but only when `related` is off. With `related: true` cross-workspace killers do run, so survivors are trustworthy. Without it, ~60% of survivors are false (killed by tests Stryker never executed).

## Integration test boundary ‚Äî run the real server in-process, mock only the datastore
This is the project-specific mechanism behind the stack-agnostic rule in arc-test
`testing-strategy.md` ("The network is not automatically the boundary") and `review.md` ¬ßD.
The boundary for a web integration test is the **datastore**, not the network.

- **Run the real BE router in-process.** The test `renderApp` builds the tRPC client on an
  in-process link that calls `appRouter.createCaller(ctx)` directly ‚Äî not the production
  `httpBatchLink`. The real router + zod validation + authorization + business logic +
  projections all execute. The web‚Üîapi HTTP call is our own internal seam, so faking its
  responses (the legacy MSW resolver pattern) fabricates our own backend ‚Äî forbidden.
- **Mock whitelist (the ONLY things a web integration test may mock):** Firestore (`getDb`),
  `firebaseAuth`, `firebaseStorage`, `firebaseMessaging`, the Anthropic translate client,
  the SMSC/OTP gateway. Everything between the component and the datastore is real.
- **Assert real BE outputs:** seed the datastore fake, drive the UI, assert the value the
  real router computes (assigned number/id, derived fields, status, localized projection) ‚Äî
  never a hand-authored fixture that duplicates that computation.
- **Narrow boundary still allowed** for behavior with no server logic (form validation,
  disabled states, file-size limits, toasts) ‚Äî render the screen without the router.
- **MSW is retained ONLY** for the thin transport-contract set the in-process boundary cannot see (headers,
  serialization, batching, HTTP-error‚ÜíTRPCError mapping ‚Äî e.g.
  `apps/web/src/shared/api/request-headers.test.ts`). It is no longer the integration boundary.

**STATUS (2026-07-19): boundary decided; in-process harness not yet built.** Until it lands:
- Legacy web tests still mock the whole backend as canned JSON via MSW
  (`apps/web/src/shared/test/trpc-server.ts` ‚Äî the `trpcMutation`/`trpcQueries` resolvers,
  and per-test `serve()` reimplementations). This is exactly the anti-pattern `review.md` ¬ßD
  now flags; it is being migrated away, not extended.
- **DB-mock mechanism: in-memory Firestore fake (chosen).** Built at
  `apps/api/src/test/firestore-fake.ts` ‚Äî a faithful fake of the Firestore Admin subset the
  API uses (collections/subcollections with auto-ids, `runTransaction` get/set/create/update/
  delete + merge, queries `==`/`array-contains-any`/`orderBy`/`startAfter(Timestamp)`/`limit`,
  `FieldValue.serverTimestamp`/`increment`, a `Timestamp` class, monotonic server timestamps).
  Control API: `fake.seed(path, data)`, `fake.reset()`, `fake.getDoc(path)`, `fake.listDocs(collPath)`.
  Wire it into the real stores with `vi.mock('../firestore', () => ({ getDb: () => fake.db, FieldValue, Timestamp }))`.
  Verified by driving the real issues store (create ‚Üí list ‚Üí changeStatus) against it in
  `apps/api/src/test/firestore-fake.test.ts`.
- **Still to build (web side):** the `renderApp` caller variant (a tRPC link calling
  `appRouter.createCaller(ctx)` instead of `httpBatchLink`), and `export const createCallerFactory = t.createCallerFactory`
  in `apps/api/src/trpc.ts`. Once done, name the concrete `renderApp` caller entry + seed helper here
  so `arc:test write` can load it, and pilot the issues vertical (create ‚Üí change status).
- Per `write.md` Step 3: until the web-side caller harness lands, a web test that needs the real
  BE has no compliant way to be written ‚Äî surface the harness as a prerequisite rather than
  fabricating the backend.

## API
- Type: tRPC (v11) ‚Äî `apps/api` exports `./src/router.ts`; web consumes via `@trpc/client` + `@trpc/tanstack-react-query`. Types flow end-to-end (no codegen needed).

## AI / Translation Pipeline (reuse for future AI features)
- `apps/api/src/translation/` ‚Äî the canonical pattern for any feature that needs an LLM call server-side. Reuse rather than re-invent:
  - `translation-client.ts` ‚Äî wraps `@anthropic-ai/sdk` (`claude-haiku-4-5`, `temperature: 0`, structured output via `zodOutputFormat`); takes `apiKey` as a param (never reaches into `process.env` itself), so it's mockable in unit tests. Exports both a live-call path (`translateDocument`/`translateText`) and a Batch-API path (`documentTranslationParams` + `parseDocumentTranslation`) sharing one output schema/guard.
  - `hash-source.ts` ‚Äî `hashSource(...parts)`: content hash (SHA-256, truncated) used as a "did the source change" guard ‚Äî prevents both re-translating unchanged content and infinite self-trigger loops on Firestore writes.
  - No-partial-write guarantee: any AI call that produces a partial/malformed result returns `null` from the client; the orchestration layer (`translate-document-fields.ts`) never writes a half-populated result. Follow this shape for any new AI-backed write path.
  - Reads never mutate stored data ‚Äî locale/variant substitution happens at the projection layer (`localized-fields.ts`), composed inside the same function that already does per-viewer projection (e.g. `parsePost`/`parseIssue`), not as a separate pass.

## Firebase Functions (Firestore triggers)
- `apps/functions` has **no test infrastructure by design** (no vitest config) ‚Äî the trigger is a thin wrapper only (read event data, call an exported `apps/api` function, write the result). All real orchestration/business logic for a trigger must live in `apps/api` (which has vitest + the `vi.mock('../firestore', ...)` convention) as a plain exported function, imported by the trigger. Don't try to add vitest/emulator infra to `apps/functions` as part of a feature ticket ‚Äî that's a separate, larger follow-up if trigger-level automated coverage is ever wanted.
- Multiple triggers needing the same secret must share one `defineSecret(...)` declaration (extract to its own module, e.g. `apps/functions/src/anthropic-key.ts`) ‚Äî declaring the same secret param twice across trigger files warns.

## File Uploads (Firebase Storage)
- Client-side upload wrappers live in `apps/web/src/shared/media/*` (e.g. `uploadAvatar`, `uploadPostMedia`) and fold the upload into the mutation's `mutationFn`, never as a separate step in the component
- Object lifecycle: a feature that replaces a stored file (avatar, post media) must clean up the object it makes obsolete ‚Äî delete the previous object after a successful update, and delete the newly-uploaded object if the following mutation fails. Otherwise superseded/orphaned objects accumulate unbounded in Storage and stale download URLs stay reachable (found in PR #61 review ‚Äî avatar re-upload/removal)
- `storage.rules`: `request.resource` is `null` on delete operations, so a `write` rule guarded by `request.resource.size`/`contentType` does not cover `delete` ‚Äî add an explicit delete clause per path whenever the feature needs to remove objects

## Raster Images (web assets)
- When adding/optimizing images in `apps/web/public/images`, resize the longest side to ~**1536px** and palette-quantize the PNG (`sharp: png({ palette: true, quality: 80, effort: 10 })`), keeping the `.png` filename so no code/design refs change. The illustrations are flat, so palette PNG is visually lossless and cuts ~85‚Äì90% weight (a batch went ~17MB ‚Üí ~2MB)
- 1536px covers iPad portrait @2x and iPhone @3x with slack ‚Äî don't go below ~1280 or above ~2048, and don't overdo it (leave iPad headroom). Keep `design/images/` masters at full resolution. Verify output visually for banding on gradient-heavy images

## Conventions
- Arrow functions only (no `function` decls); implicit return for single expressions; `const`/immutability, avoid `let`
- No type assertions (`as`, `!`) or enums; `type` over `interface`; `unknown` only at boundaries
- Array-method chaining over `for`/`for...of`; early returns over deep `if-else`/`switch`
- Naming: kebab-case files, PascalCase components, descriptive `is*/has*` flags; hooks follow `use{Entity}{Store|Data}` / `useCreate|Update|Delete{Entity}`
- Query keys via `{entity}Keys` factory (`all`/`list()`/`detail(id)`, `as const`)
- Styling: CSS modules (`*.module.scss`), design tokens in `app/tokens.scss`, class composition via `pickCss`/`cssVariables`, semantic color tokens only
- HTML: prefer semantic tags (`<section>`/`<header>`/`<button type>`/`<dialog>`/`<fieldset>`) over `<div onClick>`; mobile-only, no manual keyboard a11y
- State boundaries: server‚ÜíTanStack Query, navigation‚ÜíRouter (URL/search params), UI‚ÜíZustand; optimistic via ID-based overlays, not cache surgery

## CI
- GitHub Actions: `.github/workflows/fallow.yml` (workflow name "CI / Deploy") ‚Äî runs on `push` + `pull_request`
  - `fallow` job: `npm ci` ‚Üí `npx fallow --format sarif` ‚Üí uploads `fallow-results.sarif` artifact (report-only, `continue-on-error` ‚Äî advisory findings don't block CI)
  - `badge` job: on push to `main`, publishes a fallow health badge JSON to the `badges` branch via `peaceiris/actions-gh-pages`
  - `deploy-web` + `deploy-firebase` jobs: on push to `main`, gated on `needs: checks`. `deploy-web` builds with `VITE_BASE=/platform/` + `vars.VITE_API_URL` and publishes to GitHub Pages via `actions/deploy-pages` (Pages source = "GitHub Actions"); `deploy-firebase` runs `firebase deploy --only functions,firestore,storage` authenticated via an `authorized_user` ADC file built in-step from the `FIREBASE_TOKEN` secret (`firebase login:ci` token, used as the refresh token with the well-known firebase-tools OAuth client) and pointed at by `GOOGLE_APPLICATION_CREDENTIALS` ‚Äî the same keyless approach as `apps/api/src/dev-credentials.ts`, avoiding the deprecated `--token`/`FIREBASE_TOKEN` auth path. Each target deploys atomically (no half-deploy)
  - Node 22, npm cache
  - Secret provisioning is out-of-band and NOT checked by CI: `deploy-firebase` only validates `FIREBASE_TOKEN` exists. Any Firebase Functions secret a feature depends on (e.g. `ANTHROPIC_API_KEY` for the translation pipeline) must be set manually once via `firebase functions:secrets:set {NAME}` before the depending trigger/function works in the deployed environment ‚Äî a deploy succeeds with a missing secret, but the dependent feature silently degrades (e.g. content stays untranslated) rather than erroring. When a ticket introduces a new secret dependency, call this out explicitly in its phase result/PR description.
- Local gates: `npm run lint` (biome check), `npm run typecheck` (turbo‚Üítsc), `npm test` (vitest), `npm run fallow` (dead-code/static analysis, SARIF), `npm run lint:fsd` (Steiger FSD architecture lint, web), `npm --prefix apps/web run i18n:extract && npm --prefix apps/web run i18n:compile` (lingui catalogs ‚Äî CI's "i18n catalogs extracted" check diffs `.po` against the code, so re-extract and commit catalogs before every push, not just initial PR creation ‚Äî any subsequent fix commit that adds/removes lines shifts `#:` source-reference comments in the `.po` files and trips the drift check even when no user-facing string changed; strict compile fails on untranslated strings)
