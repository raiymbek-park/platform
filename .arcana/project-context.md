# Project Context

Generated by `/arc:project init`. Read by all Arcana skills.

## Stack
- Framework: React 19 (Vite 8) on web; tRPC 11 standalone server on api
- Language: TypeScript 6.0.3 (strict, `noUncheckedIndexedAccess`, `verbatimModuleSyntax`)
- Runtime: Node.js (tsx for the api dev/start)
- Package manager: npm 10.9.0 (workspaces) + Turborepo 2.9
- Monorepo layout: `apps/*` + `packages/*`
- Local dev quirk: `apps/functions/package.json` pins Node 22 (`engines`); the repo's `.npmrc` sets `engine-strict=true`. On a machine running a newer local Node (e.g. 24), `npm install` fails the engines check — run with `--engine-strict=false` locally. Doesn't affect CI (CI runner is Node 22).

## Testing
- Test runner: Vitest 4.1.9 (jsdom env, `@testing-library/react` + `jest-dom`)
- Mutation testing: Stryker 9.6.1 (`@stryker-mutator/vitest-runner`) — `npm run mutate` (web)
- E2E runner: CodeceptJS 4.0.8 + Playwright 1.61 — `npm run test:e2e` (via `start-server-and-test`)
- Network mocking: MSW 2.14.6
- Coverage: `@vitest/coverage-v8`
- Test directory: co-located `**/*.test.ts(x)`; e2e specs in `apps/web/src/test/**/*.e2e.ts`
- Root commands: `npm test` (vitest run), `npm run test:watch`, `npm run test:e2e`, `npm run mutate`
- Mutation testing scope is narrow: Stryker's `mutate` glob (`apps/web/stryker.config.json`) is `src/**/model/**` + `src/**/lib/**` **only**, in `apps/web` only — `ui/` components are deliberately excluded, and `apps/api` has no Stryker config at all. For any ticket whose business logic lives mostly in `apps/api` (stores, routers, trigger orchestration), `/arc:test mutate` will have little or nothing in scope to run — confidence there rests entirely on the unit/integration suite, not mutation testing. This is a known, pre-existing tooling gap; don't treat a near-empty mutate run as a phase failure or try to work around it by relocating logic into `apps/web`.

## API
- Type: tRPC (v11) — `apps/api` exports `./src/router.ts`; web consumes via `@trpc/client` + `@trpc/tanstack-react-query`. Types flow end-to-end (no codegen needed).

## AI / Translation Pipeline (reuse for future AI features)
- `apps/api/src/translation/` — the canonical pattern for any feature that needs an LLM call server-side. Reuse rather than re-invent:
  - `translation-client.ts` — wraps `@anthropic-ai/sdk` (`claude-haiku-4-5`, `temperature: 0`, structured output via `zodOutputFormat`); takes `apiKey` as a param (never reaches into `process.env` itself), so it's mockable in unit tests. Exports both a live-call path (`translateDocument`/`translateText`) and a Batch-API path (`documentTranslationParams` + `parseDocumentTranslation`) sharing one output schema/guard.
  - `hash-source.ts` — `hashSource(...parts)`: content hash (SHA-256, truncated) used as a "did the source change" guard — prevents both re-translating unchanged content and infinite self-trigger loops on Firestore writes.
  - No-partial-write guarantee: any AI call that produces a partial/malformed result returns `null` from the client; the orchestration layer (`translate-document-fields.ts`) never writes a half-populated result. Follow this shape for any new AI-backed write path.
  - Reads never mutate stored data — locale/variant substitution happens at the projection layer (`localized-fields.ts`), composed inside the same function that already does per-viewer projection (e.g. `parsePost`/`parseIssue`), not as a separate pass.

## Firebase Functions (Firestore triggers)
- `apps/functions` has **no test infrastructure by design** (no vitest config) — the trigger is a thin wrapper only (read event data, call an exported `apps/api` function, write the result). All real orchestration/business logic for a trigger must live in `apps/api` (which has vitest + the `vi.mock('../firestore', ...)` convention) as a plain exported function, imported by the trigger. Don't try to add vitest/emulator infra to `apps/functions` as part of a feature ticket — that's a separate, larger follow-up if trigger-level automated coverage is ever wanted.
- Multiple triggers needing the same secret must share one `defineSecret(...)` declaration (extract to its own module, e.g. `apps/functions/src/anthropic-key.ts`) — declaring the same secret param twice across trigger files warns.

## File Uploads (Firebase Storage)
- Client-side upload wrappers live in `apps/web/src/shared/media/*` (e.g. `uploadAvatar`, `uploadPostMedia`) and fold the upload into the mutation's `mutationFn`, never as a separate step in the component
- Object lifecycle: a feature that replaces a stored file (avatar, post media) must clean up the object it makes obsolete — delete the previous object after a successful update, and delete the newly-uploaded object if the following mutation fails. Otherwise superseded/orphaned objects accumulate unbounded in Storage and stale download URLs stay reachable (found in PR #61 review — avatar re-upload/removal)
- `storage.rules`: `request.resource` is `null` on delete operations, so a `write` rule guarded by `request.resource.size`/`contentType` does not cover `delete` — add an explicit delete clause per path whenever the feature needs to remove objects

## Raster Images (web assets)
- When adding/optimizing images in `apps/web/public/images`, resize the longest side to ~**1536px** and palette-quantize the PNG (`sharp: png({ palette: true, quality: 80, effort: 10 })`), keeping the `.png` filename so no code/design refs change. The illustrations are flat, so palette PNG is visually lossless and cuts ~85–90% weight (a batch went ~17MB → ~2MB)
- 1536px covers iPad portrait @2x and iPhone @3x with slack — don't go below ~1280 or above ~2048, and don't overdo it (leave iPad headroom). Keep `design/images/` masters at full resolution. Verify output visually for banding on gradient-heavy images

## Conventions
- Arrow functions only (no `function` decls); implicit return for single expressions; `const`/immutability, avoid `let`
- No type assertions (`as`, `!`) or enums; `type` over `interface`; `unknown` only at boundaries
- Array-method chaining over `for`/`for...of`; early returns over deep `if-else`/`switch`
- Naming: kebab-case files, PascalCase components, descriptive `is*/has*` flags; hooks follow `use{Entity}{Store|Data}` / `useCreate|Update|Delete{Entity}`
- Query keys via `{entity}Keys` factory (`all`/`list()`/`detail(id)`, `as const`)
- Styling: CSS modules (`*.module.scss`), design tokens in `app/tokens.scss`, class composition via `pickCss`/`cssVariables`, semantic color tokens only
- HTML: prefer semantic tags (`<section>`/`<header>`/`<button type>`/`<dialog>`/`<fieldset>`) over `<div onClick>`; mobile-only, no manual keyboard a11y
- State boundaries: server→TanStack Query, navigation→Router (URL/search params), UI→Zustand; optimistic via ID-based overlays, not cache surgery

## CI
- GitHub Actions: `.github/workflows/fallow.yml` (workflow name "CI / Deploy") — runs on `push` + `pull_request`
  - `fallow` job: `npm ci` → `npx fallow --format sarif` → uploads `fallow-results.sarif` artifact (report-only, `continue-on-error` — advisory findings don't block CI)
  - `badge` job: on push to `main`, publishes a fallow health badge JSON to the `badges` branch via `peaceiris/actions-gh-pages`
  - `deploy-web` + `deploy-firebase` jobs: on push to `main`, gated on `needs: checks`. `deploy-web` builds with `VITE_BASE=/platform/` + `vars.VITE_API_URL` and publishes to GitHub Pages via `actions/deploy-pages` (Pages source = "GitHub Actions"); `deploy-firebase` runs `firebase deploy --only functions,firestore,storage` authenticated by the `FIREBASE_TOKEN` secret (`firebase login:ci` token). Each target deploys atomically (no half-deploy)
  - Node 22, npm cache
  - Secret provisioning is out-of-band and NOT checked by CI: `deploy-firebase` only validates `FIREBASE_TOKEN` exists. Any Firebase Functions secret a feature depends on (e.g. `ANTHROPIC_API_KEY` for the translation pipeline) must be set manually once via `firebase functions:secrets:set {NAME}` before the depending trigger/function works in the deployed environment — a deploy succeeds with a missing secret, but the dependent feature silently degrades (e.g. content stays untranslated) rather than erroring. When a ticket introduces a new secret dependency, call this out explicitly in its phase result/PR description.
- Local gates: `npm run lint` (biome check), `npm run typecheck` (turbo→tsc), `npm test` (vitest), `npm run fallow` (dead-code/static analysis, SARIF), `npm run lint:fsd` (Steiger FSD architecture lint, web), `npm --prefix apps/web run i18n:extract && npm --prefix apps/web run i18n:compile` (lingui catalogs — CI's "i18n catalogs extracted" check diffs `.po` against the code, so re-extract and commit catalogs before every push, not just initial PR creation — any subsequent fix commit that adds/removes lines shifts `#:` source-reference comments in the `.po` files and trips the drift check even when no user-facing string changed; strict compile fails on untranslated strings)
