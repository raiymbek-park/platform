## Testing
- Test runner: Vitest 4.1.9 (jsdom env, `@testing-library/react` + `jest-dom`)
- Mutation testing: Stryker 9.6.1 (`@stryker-mutator/vitest-runner`) — `npm run mutate` (one root config, `stryker.config.json`)
- E2E runner: CodeceptJS 4.0.8 + Playwright 1.61 — `npm run test:e2e` (via `start-server-and-test`)
- Network mocking: MSW 2.14.6
- Coverage: `@vitest/coverage-v8`
- Test directory: co-located `**/*.test.ts(x)`; e2e specs in `apps/web/src/test/**/*.e2e.ts`
- Root commands: `npm test` (vitest run), `npm run test:watch`, `npm run test:e2e`, `npm run mutate`
- Mutation testing runs from **one root config** (`stryker.config.json`) over the whole vitest workspace. Scope: `apps/web/src/**/model|lib/**` + `apps/api/src/**` + `packages/shared/src/**`. `packages/ui` and web `ui/` components are deliberately excluded (they are covered behaviorally, per the testing strategy).
- **Always scope a run to the ticket's 2–3 files** (`npx stryker run --mutate '<path>'`), never bare `npm run mutate`. A full sweep is ~580 mutants and will not finish in a useful time; a scoped run on the file that holds the ticket's rules is ~2–3 minutes. This is the single most-violated rule here — an unscoped sweep is what makes the phase look worthless.
- **Derive the scope from git instead of hand-listing files.** Compute the branch's changed files and feed them to `--mutate` (bash):
  ```bash
  FILES=$(git diff --name-only --diff-filter=d main...HEAD \
    | grep -E '^(packages/shared/src|apps/api/src|apps/web/src/.*(model|lib))/.*\.ts$' \
    | grep -vE '\.(test|spec|gen|d)\.ts$' | paste -sd,)
  npx stryker run --mutate "$FILES"
  ```
  `main...HEAD` (three dots) diffs from the branch point, so the run covers exactly what the branch added. This is StrykerJS's equivalent of .NET's `--diff` — **StrykerJS has no `--since` flag**. `incremental: true` (already set) is the complementary auto-diff: it reuses the prior report so repeat runs re-test only changed code/tests. Use the git recipe for a deterministic branch-scoped run; incremental for cheap local re-runs.
- **`vitest.related: true` is load-bearing, not an optimization.** Vitest walks the import graph and runs only the tests that actually reach the mutated file — *including tests in other workspaces*. Measured on `packages/shared/src/validation-schemas/resident.ts` (148 mutants): shared's own suite alone → 71.62%, 38 survivors, 29s; whole monorepo suite (`related: false`) → 87.16%, 15 survivors, **23m33s**; `related: true` → 88.51%, 13 survivors, **2m31s**. Same signal as the exhaustive run, ~9× faster.
- **Business rules live in `packages/shared`, not in `apps/web`.** `apps/web/src/**/lib/*` holds thin re-exports (`carrier-warning.ts`, `phone.ts`); every AC-critical rule — apartment ranges, name bounds, phone validity, E.164 normalization, carrier prefixes — is in `packages/shared/src/validation-schemas/resident.ts`. Point a mutation run at the rule, not at its adapter.
- **In a monorepo a survivor is a lead, not a verdict** — but only when `related` is off. With `related: true` cross-workspace killers do run, so survivors are trustworthy. Without it, ~60% of survivors are false (killed by tests Stryker never executed).

## Integration test boundary — run the real server in-process, mock only the datastore
This is the project-specific mechanism behind the stack-agnostic rule in arc-test
`testing-strategy.md` ("The network is not automatically the boundary") and `review.md` §D.
The boundary for a web integration test is the **datastore**, not the network. **Built and in
use** — ADR 017.

- **How (web side).** `renderAppWithServer(path, { uid })` (`apps/web/src/shared/test/render-app-server.tsx`)
  renders the real app and registers one MSW handler `http.all('/trpc/*') → fetchRequestHandler(appRouter)`,
  so **every** tRPC call (both the provider client and code that uses the singleton `trpcClient`
  directly) runs the real router in-process — real zod validation, authorization, business logic,
  projections. The web↔api HTTP call is our own internal seam; faking its responses fabricates our
  own backend — forbidden.
- **DB = the only mock, via dependency injection (NOT module mocking).** Vitest externalizes the
  `@raiymbek-park/api` package, so `vi.mock('firebase-admin')` cannot reach it from a web test.
  Instead inject the in-memory fake through the DI seam in `apps/api/src/firestore.ts`. In a web
  test: import `{ fake, injectFake, resetFirestore }` from `@raiymbek-park/api/testing`; in
  `beforeEach` call `firebaseAuth.signIn(); fake.reset(); injectFake();` then
  `fake.seed('residents/<uid>', { role, … })`; `afterEach(resetFirestore)`. `injectFake()` makes
  `getDb()` return the fake and swaps `FieldValue`, so **firebase-admin never initializes — tests
  never touch production**.
- **Firebase Auth + SMS (onboarding / auth flows):** `injectFake` also injects a fake Firebase Auth
  admin — `getUserByPhoneNumber`/`createUser`/`createCustomToken` via `authFake` from
  `@raiymbek-park/api/testing` (`authFake.seedUser(phone, uid)` for a returning user, `authFake.reset()`).
  For the SMS+code, set `process.env.OTP_TEST_MODE = 'true'` and use the test phone `+77781234455`
  (→ fixed code `123456` via `test-codes.ts`) so the real otp router runs (reserveSend / verifyAttempt /
  getOrCreateUid) with no real SMS. Verified in `apps/api/src/test/auth-fake.test.ts`. Auth injection is
  cleared by `resetFirestore()`. Note: `createContext` injects a fixed `uid`, so for the post-verify
  registration step render with the uid the flow produces (OTP procedures ignore `uid`) and align it with
  the client `firebaseAuth` mock.
- **Fake:** `apps/api/src/test/firestore-fake.ts` — faithful subset (collections/subcollections with
  auto-ids, `runTransaction` and direct `doc().get/set/update/delete/create` + merge, queries
  `==`/`array-contains-any`/`orderBy`/`startAfter`/`limit`, `FieldValue.serverTimestamp`/`increment`,
  real `Timestamp` via `fromMillis`, monotonic clock). Control API: `fake.seed/reset/getDoc/listDocs`.
  It is **load-bearing and grows per vertical** — a store that calls an unimplemented method fails; add
  the method faithfully (e.g. `doc().create()` was added for the posts vertical).
- **One integration file per screen (variant A).** A single harness test (`*.server.test.tsx`) covers
  a screen's client-side behavior (validation, disabled, limits) AND its backend-touching behavior.
  Do NOT split a screen into a "narrow" and a "wide" file. Assign top-down: cover through the page/main
  form; drop lower (component/unit/pure function) only for what the page can't reach.
- **Error / empty / loading** cases use a canned response at the boundary (per §D carve-out): add a
  runtime MSW handler AFTER `renderAppWithServer` (e.g. `trpcServer.use(trpcMutationError('posts.create'))`)
  — it wins for that procedure while everything else hits the real router. This is testing the UI's
  reaction, not fabricating a success.
- **Mock whitelist (the ONLY things a web integration test may mock):** the datastore (via `injectFake`),
  `firebaseAuth`, `firebaseStorage`, `firebaseMessaging`, the Anthropic translate client, the SMSC/OTP
  gateway. Everything between the component and the datastore is real.
- **Assert real BE outputs:** seed the fake, drive the UI, assert both the UI and the value the real
  router computed (`fake.listDocs/getDoc` → assigned number/id, derived fields, status, projection) —
  never a hand-authored fixture that duplicates that computation.
- **API-side tests** (`apps/api/**`, node env, not externalized) keep using
  `vi.mock('../firestore', () => ({ getDb: () => fake.db, FieldValue, Timestamp }))` — that path still works.
- **MSW is retained** as the transport for the harness itself, and for the thin transport-contract set
  the in-process path cannot see (headers, serialization, batching, HTTP-error→TRPCError mapping — e.g.
  `apps/web/src/shared/api/request-headers.test.ts`).
- Naming/lint: harness files are plain `*.test.tsx` (no special suffix). They import
  `render-app-server` **directly** (`@/shared/test/render-app-server`), NOT via the `@/shared/test`
  barrel — a barrel re-export would eagerly pull the heavy `appRouter` (firebase-admin + Anthropic SDK)
  into every web test that touches the barrel, thrashing the parallel run; the direct import confines
  that load to harness files only. `apps/web/steiger.config.ts` excludes all test files from FSD, so the
  direct import doesn't trip `fsd/no-public-api-sidestep`.
- **Language: harness tests are authored in English.** `renderAppWithServer` activates the English lingui
  catalog and sets `createContext` locale to `en`, so the UI renders in English — assert English chrome
  (buttons/toasts/labels/category/status names, taken from `src/shared/i18n/locales/en/messages.po`), and
  write sample user content in English too. Test names are English only — no localized UI labels quoted in
  the title, and no filler restating what the harness already guarantees ("through the real backend", "it is
  stored", "runs the real router"); name the user-visible behaviour. (Narrow non-harness tests still render
  the Russian default until they migrate — `vitest.setup` resets the locale to `ru` per test.)

**Known limitation.** `createContext` injects `uid` directly, so **role-based authorization is testable**
(seed the resident's `role`; the real `getRole` runs), but **server-side identity rejection** (missing/invalid
token → `UNAUTHORIZED`) is NOT reachable through the harness. Cover such 401 paths at the api layer, or fall
back to a canned boundary error for the UI-reaction half.

**Test tiers (the layered model).**
- **Unit** — pure functions (validation, projection, tokenization, formatting). `npm test`.
- **Integration (in-memory fake)** — the web harness above, plus api store tests that seed the fake, run the
  real code, and assert the read-back result. Fast, parallel, `npm test`. This is the bulk.
- **Emulator (real Firestore)** — `apps/api/src/infra-2.test.ts`, gated by `describe.skipIf(!FIRESTORE_EMULATOR_HOST)`,
  run via `npm --prefix apps/api run test:emulator` (`npx firebase-tools@15 emulators:exec --only firestore …`,
  **needs Java 21**). Reserved for what a fake can't prove — transaction atomicity/concurrency, real `increment`/
  `merge`, query/index correctness, and `firestore.rules` (via `@firebase/rules-unit-testing`). Each test does
  `clearAll` + seed (so order-independent); run serially (or per-`projectId`), never in the fast parallel pool.
  Wired into CI as a step in the `checks` job (with a Java 21 setup); NOT part of the default `npm test`. Keep it THIN.

Assign top-down (arc-test `testing-strategy.md`): cover a behaviour at the highest tier that reaches it — a web
harness test that drives the real router+store already covers that server logic, so don't duplicate it in an api
store/router test. Keep api tests for server-only flows (digests, triggers), UI-unreachable branches (non-author
authorization, identity gates, legacy/malformed data), pure utilities, and the emulator-grade guarantees above.

## AI / Translation Pipeline (reuse for future AI features)
- `apps/api/src/translation/` — the canonical pattern for any feature that needs an LLM call server-side. Reuse rather than re-invent:
  - `translation-client.ts` — wraps `@anthropic-ai/sdk` (`claude-haiku-4-5`, `temperature: 0`, structured output via `zodOutputFormat`); takes `apiKey` as a param (never reaches into `process.env` itself), so it's mockable in unit tests. Exports both a live-call path (`translateDocument`/`translateText`) and a Batch-API path (`documentTranslationParams` + `parseDocumentTranslation`) sharing one output schema/guard.
  - `hash-source.ts` — `hashSource(...parts)`: content hash (SHA-256, truncated) used as a "did the source change" guard — prevents both re-translating unchanged content and infinite self-trigger loops on Firestore writes.
  - No-partial-write guarantee: any AI call that produces a partial/malformed result returns `null` from the client; the orchestration layer (`translate-document-fields.ts`) never writes a half-populated result. Follow this shape for any new AI-backed write path.
  - Reads never mutate stored data — locale/variant substitution happens at the projection layer (`localized-fields.ts`), composed inside the same function that already does per-viewer projection (e.g. `parsePost`/`parseIssue`), not as a separate pass.

## Firebase Functions (Firestore triggers)
- `apps/functions` has **no test infrastructure by design** (no vitest config) — the trigger is a thin wrapper only (read event data, call an exported `apps/api` function, write the result). All real orchestration/business logic for a trigger must live in `apps/api` (which has vitest + the `vi.mock('../firestore', ...)` convention) as a plain exported function, imported by the trigger. Don't try to add vitest/emulator infra to `apps/functions` as part of a feature ticket — that's a separate, larger follow-up if trigger-level automated coverage is ever wanted.
- Multiple triggers needing the same secret must share one `defineSecret(...)` declaration (extract to its own module, e.g. `apps/functions/src/anthropic-key.ts`) — declaring the same secret param twice across trigger files warns.

## File Uploads (Firebase Storage)
- Client-side upload wrappers live in `apps/web/src/shared/media/*` (e.g. `uploadAvatar`, `uploadPostMedia`) and fold the upload into the mutation's `mutationFn`, never as a separate step in the component
- Object lifecycle: a feature that replaces a stored file (avatar, post media) must clean up the object it makes obsolete — delete the previous object after a successful update, and delete the newly-uploaded object if the following mutation fails. Otherwise superseded/orphaned objects accumulate unbounded in Storage and stale download URLs stay reachable (found in PR #61 review — avatar re-upload/removal)
- `storage.rules`: `request.resource` is `null` on delete operations, so a `write` rule guarded by `request.resource.size`/`contentType` does not cover `delete` — add an explicit delete clause per path whenever the feature needs to remove objects

## Raster Images (web assets)
- When adding/optimizing images in `apps/web/public/images`, resize the longest side to ~**1536px** and palette-quantize the PNG (`sharp: png({ palette: true, quality: 80, effort: 10 })`), keeping the `.png` filename so no code/design refs change. The illustrations are flat, so palette PNG is visually lossless and cuts ~85–90% weight (a batch went ~17MB → ~2MB)
- 1536px covers iPad portrait @2x and iPhone @3x with slack — don't go below ~1280 or above ~2048, and don't overdo it (leave iPad headroom). Keep `design/images/` masters at full resolution. Verify output visually for banding on gradient-heavy images

## Conventions
- Arrow functions only (no `function` decls); implicit return for single expressions; `const`/immutability, avoid `let`
- No type assertions (`as`, `!`) or enums; `type` over `interface`; `unknown` only at boundaries
- Array-method chaining over `for`/`for...of`; early returns over deep `if-else`/`switch`
- Naming: kebab-case files, PascalCase components, descriptive `is*/has*` flags; hooks follow `use{Entity}{Store|Data}` / `useCreate|Update|Delete{Entity}`
- Query keys via `{entity}Keys` factory (`all`/`list()`/`detail(id)`, `as const`)
- Styling: CSS modules (`*.module.scss`), design tokens in `app/tokens.scss`, class composition via `pickCss`/`cssVariables`, semantic color tokens only
- HTML: prefer semantic tags (`<section>`/`<header>`/`<button type>`/`<dialog>`/`<fieldset>`) over `<div onClick>`; mobile-only, no manual keyboard a11y; collapse single-child wrappers whose styles merge onto the parent (merge-up test) — no wrapper for anticipated-but-absent structure
- State boundaries: server→TanStack Query, navigation→Router (URL/search params), UI→Zustand; optimistic via ID-based overlays, not cache surgery

## CI
- GitHub Actions: `.github/workflows/fallow.yml` (workflow name "CI / Deploy") — runs on `push` + `pull_request`
  - `fallow` job: `npm ci` → `npx fallow --format sarif` → uploads `fallow-results.sarif` artifact (report-only, `continue-on-error` — advisory findings don't block CI)
  - `badge` job: on push to `main`, publishes a fallow health badge JSON to the `badges` branch via `peaceiris/actions-gh-pages`
  - `deploy-web` + `deploy-firebase` jobs: on push to `main`, gated on `needs: checks`. `deploy-web` builds with `VITE_BASE=/platform/` + `vars.VITE_API_URL` and publishes to GitHub Pages via `actions/deploy-pages` (Pages source = "GitHub Actions"); `deploy-firebase` runs `firebase deploy --only functions,firestore,storage` authenticated via an `authorized_user` ADC file built in-step from the `FIREBASE_TOKEN` secret (`firebase login:ci` token, used as the refresh token with the well-known firebase-tools OAuth client) and pointed at by `GOOGLE_APPLICATION_CREDENTIALS` — the same keyless approach as `apps/api/src/dev-credentials.ts`, avoiding the deprecated `--token`/`FIREBASE_TOKEN` auth path. Each target deploys atomically (no half-deploy)
  - Node 22, npm cache
  - Secret provisioning is out-of-band and NOT checked by CI: `deploy-firebase` only validates `FIREBASE_TOKEN` exists. Any Firebase Functions secret a feature depends on (e.g. `ANTHROPIC_API_KEY` for the translation pipeline) must be set manually once via `firebase functions:secrets:set {NAME}` before the depending trigger/function works in the deployed environment — a deploy succeeds with a missing secret, but the dependent feature silently degrades (e.g. content stays untranslated) rather than erroring. When a ticket introduces a new secret dependency, call this out explicitly in its phase result/PR description.
- Local gates: `npm run lint` (biome check), `npm run typecheck` (turbo→tsc), `npm test` (vitest), `npm run fallow` (dead-code/static analysis, SARIF), `npm run lint:fsd` (Steiger FSD architecture lint, web), `npm --prefix apps/web run i18n:extract && npm --prefix apps/web run i18n:compile` (lingui catalogs — CI's "i18n catalogs extracted" check diffs `.po` against the code, so re-extract and commit catalogs before every push, not just initial PR creation — any subsequent fix commit that adds/removes lines shifts `#:` source-reference comments in the `.po` files and trips the drift check even when no user-facing string changed; strict compile fails on untranslated strings)
